## 3. 核心业务目标

剔除旧的 legacy 业务机采集客户端，重构为订单整理系统官方配套工具“打印组件信息采集”。

打印组件信息采集部署在业务机上，只负责：

- 读取业务机本地打印组件数据。
- 完整回传面单原始信息。
- 上报任务 ID、组件 rowid、原始 payload、打印原文等排查字段。

打印组件信息采集不负责：

- 面单识别。
- 数据筛选。
- 五要素解析。
- 订单整理。
- Excel 生成。
- 鞋款规则维护。
- 图片关系维护。

最高优先级原则：

> 宁可多上传，不允许漏掉任何 task_id 或 component_rowid。

## 4. 业务机采集助手定位

### 4.1 官方名称

打印组件信息采集

### 4.2 内部名称

OrderCollectorAgent

### 4.3 源码定位

源码属于订单整理系统内置插件体系，必须放入：

```text
src/plugins/collector_agent/
```

### 4.4 运行定位

运行时是发给业务机使用的独立轻量 EXE、托盘程序或后台采集服务。

### 4.5 业务定位

OrderCollectorAgent 是订单整理系统官方配套采集端：

- 不是临时脚本。
- 不是 legacy 工具。
- 不是完整订单整理客户端。
- 与订单整理系统同仓库、同版本体系、同发布规范。
- 但需要单独构建为业务机运行端。

### 4.6 为什么需要独立 EXE

业务机采集助手需要读取业务机本地打印组件数据库。Web 页面无法直接稳定读取业务机本机程序目录，所以必须在业务机安装本地采集助手。

### 4.7 禁止定位

- 不得继续叫 legacy。
- 不得放在 scripts/collector_client_legacy/。
- 不得作为临时脚本散落。
- 不得把客户机工具做成完整订单整理系统。
- 不得把采集逻辑只做在 Web 前端。

## 5. 版本生命周期策略

从 v7.9.3 开始，每次订单整理系统发布新版本时，都必须检查并声明业务机采集助手的版本状态。

### 5.1 发布目录规则

每个版本目录 versions/vX.Y.Z/ 必须包含主系统产物信息和采集助手产物信息。

建议路径：

```text
versions/vX.Y.Z/bin/
versions/vX.Y.Z/bin/打印组件信息采集_vX.Y.Z.zip
versions/vX.Y.Z/bin/打印组件信息采集_vX.Y.Z.exe
versions/vX.Y.Z/release_manifest.json
versions/vX.Y.Z/tests/report.log
```

### 5.2 当采集助手代码发生变化

必须执行：

- 更新 OrderCollectorAgent 版本号。
- 重新构建业务机采集助手 EXE。
- 执行采集助手专项测试。
- 在版本说明中写明业务机是否需要升级。

### 5.3 当采集助手代码没有变化

处理规则：

- 不强制重打业务机 EXE。
- release_manifest.json 中必须记录本版本兼容的 agent 版本。
- Web 后端必须能识别旧 agent 是否兼容。
- 版本说明中明确写：业务机采集助手无需升级，继续使用指定兼容版本。

### 5.4 协议兼容字段

采集助手上报必须包含：

- agent_version
- protocol_version
- client_id
- machine_name
- machine_label
- component_status
- last_seen

服务端响应建议包含：

- server_version
- protocol_version
- min_supported_agent_version
- latest_agent_version
- upgrade_required
- upgrade_message
- download_url

规则：

只要 Web 接口协议或上传字段发生不兼容变化，必须提升 protocol_version，并提示业务机升级采集助手。

### 5.5 升级策略

默认不自动强制升级业务机工具，只在 Web 管理端提示版本状态。

强制升级条件：

- 上传协议不兼容。
- 存在严重漏采风险。
- 旧版本无法识别任务 ID。
- 旧版本存在数据丢失缺陷。

Web 管理页面必须显示每台业务机：

- agent_version
- protocol_version
- 是否需要升级
- 最新安装包下载入口

## 9. 阶段四：不漏任务 ID / rowid 的硬性规则

### 9.1 目标

确保不漏掉任何打印任务 ID，这是本版本最高优先级验收项。

### 9.2 硬性规则

不得因为以下情况丢弃任务：

- 打印文本为空。
- JSON 解析失败。
- printXML 解析失败。
- document 为空。
- 文本重复。
- 打印信息相同。
- 商品信息为空。
- 未识别出业务字段。

### 9.3 禁止行为

- 不得仅按打印信息文本去重。
- 不得只上传识别成功的记录。
- 不得只上传有商品信息的记录。
- 不得覆盖同 task_id 下的多条 document 记录。
- 不得将相同 task_id 的不同 rowid 合并为一条。
- 不得将相同 print_text_raw 的不同 rowid 合并为一条。

### 9.4 最低记录保证

打印组件数据库 task 表中的每一行 rowid，只要进入本次批次范围，至少必须生成一条上传记录。

### 9.5 允许的去重规则

允许防止重复上传同一数据库行，但去重键必须包含：

- component_db_id
- component_rowid
- task_id
- document_id
- source_record_index

禁止使用以下字段作为唯一去重依据：

- print_text_raw
- normalized_print_text
- task_id_only
- document_id_only

规则：

允许防止重复上传同一数据库行，但禁止因为文本相同、task_id 相同或解析结果相同而合并掉不同 rowid。

### 9.6 游标规则

本地需要保存每个 component_db_id 的 last_uploaded_rowid 和 pending 状态。

批次开始：

- 记录每个组件当前最大 rowid 作为 baseline。

批次结束：

- 上传 rowid 大于 baseline 的所有任务行。

离线场景：

- 如果 stop 时无法连接 Web，必须把本次待上传记录写入 pending_uploads。
- 恢复连接后继续上传。

确认规则：

- 只有 Web 后端确认 accepted 后，才允许更新 last_uploaded_rowid 或清理 pending 文件。

### 9.7 上传记录必填字段

每条上传记录必须尽量包含：

- client_id
- machine_name
- machine_label
- agent_version
- protocol_version
- batch_id
- component_name
- component_db_id
- component_db_path
- component_rowid
- task_id
- document_id
- task_time
- source_record_index
- raw_msg_json
- raw_document_json
- print_text_raw
- extract_status
- created_at

### 9.8 异常场景兜底行为

#### msg JSON 解析失败

仍必须上传记录：

- 上传原始 msg 字符串到 raw_msg_json 或 raw_error_text。
- task_id 使用数据库字段 taskID。
- extract_status = raw_json_parse_failed。

#### documents 为空

仍必须上传一条记录：

- 保留完整 payload。
- extract_status = no_documents。

#### printXML 无法解析

仍必须上传记录：

- 保留完整 raw_document_json。
- extract_status = print_xml_parse_failed。

#### 没有提取到任何文字

仍必须上传任务记录：

- print_text_raw 使用占位提示。
- raw_document_json 保存原始 document。

## 10. 阶段五：Web 后端接口

### 10.1 目标

Web 后端作为业务机上线注册、轮询、上传、版本管理和原文查看中心。

### 10.2 新增或重构接口

#### POST /api/collector/register

用途：业务机工具上线后自动向 Web 后端注册，获取 agent_token。

调用方：CollectorAgent。

#### POST /api/collector/poll

用途：业务机工具轮询 Web，汇报在线状态、组件状态、版本信息，并获取 start、stop、idle、upgrade 指令。

调用方：CollectorAgent。

#### POST /api/collector/upload

用途：业务机工具上传批次打印任务原文。

调用方：CollectorAgent。

#### GET /api/collector/agents

用途：Web 查询业务机列表、在线状态、版本、最近上传。

调用方：Web UI。

#### GET /api/collector/records

用途：Web 查询上传原文记录。

调用方：Web UI。

#### GET /api/collector/version-info

用途：业务机工具检查当前服务端版本、最低兼容版本、最新下载地址。

调用方：CollectorAgent。

### 10.3 鉴权规则

- 上线注册成功后生成 agent_token。
- agent_token 保存在客户机 config 目录。
- poll 和 upload 必须携带 token。
- 服务端校验 token 后才能接受上传。
- 不得匿名上传。

### 10.4 与现有面单接口兼容

- 现有 /api/waybill/start 和 /api/waybill/stop 可继续作为 Web 用户操作入口。
- 后端内部可以将 start/stop 状态转发给 /api/collector/poll。
- 旧 /api/waybill/agent/poll 和 /api/waybill/agent/upload 如保留，必须标记为兼容层，不再作为新主接口。
- 新代码优先使用 /api/collector/* 命名。
