# V7.8.0 五要素模块方案

## 核心结论

系统主流程固定为：

```text
原始数据
  -> 五要素识别
  -> 鞋款分类
  -> 鞋款档口绑定
  -> 鞋款规格图片绑定
  -> 输出整理文档
```

## 五要素

| 字段 | 说明 |
| --- | --- |
| 鞋款 | 真正的鞋款主名，例如 `4.0`、`5.0`、`Cloudtilt`、`one帆布kw` |
| 规格 | 同一鞋款下的颜色、代数、货号、版本等发货差异 |
| 尺码 | 发货尺码 |
| 数量 | 发货数量 |
| 备注 | 买家留言、卖家备注、人工补充、异常修正 |

## 模块边界

### 1. 五要素识别模块

文件：`five_field_normalizer.py`

职责：

- 接收已经按模板拆出的原始单项。
- 输出固定字段：`鞋款 / 规格 / 尺码 / 数量 / 备注`。
- 同时保留旧字段镜像：`商品简称 / 货品标题`，让旧档口和图片逻辑逐步迁移，不一次性打断。

### 2. 鞋款档口模块

职责：

- 只维护 `鞋款分类 -> 档口` 的绑定。
- 不负责识别规格、尺码、图片。
- 一个鞋款分类默认只对应一个发货档口。

### 3. 鞋款规格图片模块

职责：

- 维护 `鞋款分类 -> 规格 -> 图片`。
- 同一个鞋款下面允许多个规格。
- 规格可以有别名，供用户处理 `白黑 / 白色黑 / Cloud白黑` 这类写法。
- 图片关系只在五要素完成后匹配，不直接解析原始订单长标题。

## 当前落地状态

- 已新增 `five_field_normalizer.py`。
- `order_core.read_by_template()` 已改为先生成五要素标准行。
- `order_core.build_result()` 已优先消费五要素里的 `鞋款 / 规格 / 尺码 / 数量 / 备注`。
- 输出文档已把原 `商品简称` 列调整为 `鞋款`。
- 旧字段仍保留在内部，保证 V7.7.1 旧逻辑可以逐步迁移。

## 后续维护重点

1. 给五要素识别模块增加识别报告：空鞋款、异常尺码、规格缺失、备注拆分失败。
2. 管理端新增“鞋款下规格管理”，让用户维护同鞋款下的规格、别名和图片。
3. 逐步把数据内部旧名 `category_rules`、`stall_map` 迁移成更清楚的鞋款语义。
4. Web 后端移除历史重复的旧整理函数，只保留 `order_core.generate_order_file()` 一条路径。
