# 项目整改评估

## 当前结论

V7.9.3 的整改方向已经从旧独立采集客户端切换为官方“打印组件信息采集”。

已完成的结构性调整：

- 主系统源码继续按 `src/core/`、`src/ui/`、`src/utils/`、`src/tests/` 管理。
- 打印组件信息采集源码集中在 `src/plugins/collector_agent/`。
- Web 后端通过 `/api/collector/*` 管理上线注册、轮询、上传、业务机状态、原文记录和版本信息。
- 构建产物进入 `versions/vX.Y.Z/`，业务机只领取“打印组件信息采集” EXE 或 zip。

## 关键风险控制

- 打印组件信息采集不在业务机端识别、筛选、合并或丢弃打印任务。
- 每个进入批次范围的 `component_rowid` 至少生成一条上传记录。
- JSON 解析失败、documents 为空、printXML 失败、空打印文本、重复文本和重复 task_id 都必须保留可追溯原文。
- 上传失败进入本地 pending 队列，服务端确认 accepted 后才更新游标。

## 后续检查

每次发布新版本时，需要检查 `release_manifest.json`，明确本版本兼容的采集端版本，以及业务机是否需要升级。
